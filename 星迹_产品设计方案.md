# 星迹 (StarTrace) - 产品设计方案 v1.1

---

## 一、产品定位与愿景

### 一句话定位
**星迹**是一款以"星系"为空间隐喻的灵感管理与AI故事创作工具——碎片化灵感散落为星，AI生成的故事聚为星云，用户在可视化星图中拖拽探索自己的思维宇宙。

### 核心价值主张
- **捕捉不等待**：灵感闪现时，3秒内完成记录，不打断心流
- **可视化的思维空间**：灵感不再是扁平列表，而是可拖拽、可缩放、可感知关联的立体星系
- **AI为你续写**：碎片不再是终点——用户选择碎片，AI生成专属于自己的故事，故事成为星系中最亮的那颗星
- **数据主权在用户**：核心功能完全离线，AI模型由用户自己配置，故事上传完全自愿

### 目标用户画像
| 维度 | 描述 |
|------|------|
| 核心用户 | 创意工作者、写作者、设计师、产品经理等需要频繁捕捉灵感碎片并期望将其转化为完整产物的脑力劳动者 |
| 使用场景 | 通勤途中、睡前灵感、会议间隙——任何"想记下来再说"的瞬间 |
| 核心痛点 | 灵感碎片越积越多但不整理（或不知道如何整理），缺乏从碎片到成品的转化路径 |
| 情感诉求 | 看到自己的思维成果可视化地"生长"，获得创造的成就感和掌控感 |

## 二、功能规格（PRD风格）

### 2.1 问题陈述

灵感型用户面临三个渐进式痛点：
1. **捕获断层**：灵感转瞬即逝，现有的备忘录/笔记工具操作路径太长，丧失"即刻记录"的冲动
2. **积累焦虑**：碎片越积越多，但扁平列表让用户看不到全貌、感知不到关联，产生"记了也白记"的挫败感
3. **转化缺失**：碎片到成品的鸿沟始终存在——用户缺乏从碎片到完整叙事作品的中间手段，碎片最终被遗忘

### 2.2 Goals

| # | 目标 | 衡量方式 |
|---|------|---------|
| G1 | 灵感碎片记录时间 < 3秒（从打开App到完成保存） | 埋点计时 |
| G2 | 用户留存率：D7 > 40%，D30 > 25% | 行为埋点 |
| G3 | 月均碎片→故事转化率 > 15%（每100条碎片至少生成15个故事） | 数据库统计 |
| G4 | 星系视图单次使用时长中位数 > 3分钟 | 埋点计时 |
| G5 | App Store评分 > 4.5 | ASO监测 |

### 2.3 Non-Goals（v1明确不做）

| # | Non-Goal | 原因 |
|---|----------|------|
| N1 | 多设备同步 | v1离线为主，云端同步涉及复杂冲突解决，放到v2 |
| N2 | 协作共创/多人编辑 | 与"个人思维空间"定位冲突，暂不进入 |
| N3 | AI生成图片/配图 | 增加LLM调用复杂度，且偏离文字叙事核心 |
| N4 | 内置推荐/社交Feed算法 | 故事广场v1仅简单时间排序，不引入推荐模型 |
| N5 | 导出PDF/Markdown外的格式 | 聚焦核心输出格式 |
| N6 | 碎片自动分类/标签推荐（AI驱动） | v1只做手动标签，避免离线场景AI依赖过重 |

### 2.4 用户故事

#### 灵感捕获模块

| 优先级 | 用户故事 |
|--------|---------|
| P0 | 作为灵感记录者，我希望在3秒内完成一次碎片录入（打开→输入→保存），以便不打断当前的思绪 |
| P0 | 作为常记录的用户，我希望自动为每条碎片添加时间戳，以便追溯灵感的来源时刻 |
| P1 | 作为组织型用户，我希望为碎片添加自定义标签，以便按主题分类管理 |
| P1 | 作为感性用户，我希望标注碎片的情绪基调（兴奋/平静/困惑/惊叹），以便后续按情绪筛选 |
| P2 | 作为重度用户，我希望手动关联相关碎片，以便标记这些碎片之间有内在联系 |

#### 星系可视化模块

| 优先级 | 用户故事 |
|--------|---------|
| P0 | 作为用户，我希望看到所有碎片以星点形式散布在星系视图中，以便直观感知自己灵感积累的"规模感" |
| P0 | 作为用户，我希望拖拽和缩放在星系中自由浏览，以便像探索星空一样发现自己的灵感 |
| P0 | 作为用户，我希望点击某个星点查看该碎片的完整内容，以便快速回顾当时的灵感 |
| P1 | 作为用户，我希望星系中不同类型的内容（碎片/故事）有视觉区分（如星点 vs 星云），以便快速定位 |
| P1 | 作为用户，我希望看到碎片与故事之间的关联连线，以便理解灵感是如何演化成故事的 |
| P1 | 作为用户，我希望按时间或标签筛选星系中显示的内容，以便聚焦特定主题 |
| P2 | 作为用户，我希望星系视图有"星座"自动聚合功能，将相似标签的碎片在视觉上聚类 |

#### 故事生成模块

| 优先级 | 用户故事 |
|--------|---------|
| P0 | 作为用户，我希望能从星系中多选碎片作为故事素材，以便AI基于我的真实灵感生成故事 |
| P0 | 作为用户，我希望自定义故事的篇幅（短篇/中篇/长篇），以便控制生成内容的长度 |
| P0 | 作为用户，我希望选择故事风格（科幻/奇幻/现实/散文/诗歌等），以便获得符合预期的叙事风格 |
| P0 | 作为用户，我希望配置自己的大模型URL和Key（在设置界面），以便使用自己拥有的AI服务 |
| P1 | 作为用户，我希望对生成的故事进行多轮对话式润色，以便迭代优化直到满意 |
| P1 | 作为用户，我希望生成的故事自动存入星系并标记为"星云"类型，以便在星系中看到它的位置 |
| P1 | 作为用户，我希望故事详情页展示它使用了哪些碎片作为素材，以便追溯创作来源 |
| P2 | 作为用户，我希望保存多个故事版本并支持版本切换，以便不丢失任何一版好构思 |

#### 用户系统模块

| 优先级 | 用户故事 |
|--------|---------|
| P0 | 作为新用户，我希望能用用户名+密码注册账号（含防人机验证），以便创建专属账户 |
| P0 | 作为已有用户，我希望能用用户名+密码登录，以便进入自己的灵感空间 |
| P0 | 作为用户，我希望在"我的"页面查看个人信息和统计数据，以便了解自己的创作情况 |
| P1 | 作为创作者，我希望将生成的故事上传到公开故事广场，以便其他用户阅读我的作品 |
| P1 | 作为读者，我希望浏览公开故事广场，以便发现其他用户基于灵感碎片创作的精彩故事 |
| P1 | 作为用户，我希望管理自己已公开的故事（下架/重新编辑），以便控制对外展示的内容 |
| P2 | 作为用户，我希望修改密码和完善个人资料，以便管理账户安全 |

#### 设置模块

| 优先级 | 用户故事 |
|--------|---------|
| P0 | 作为用户，我希望在设置中配置大模型的API URL和Key，以便App能调用我的LLM服务 |
| P0 | 作为用户，我希望测试LLM连接状态，以便确认配置是否正确 |
| P1 | 作为用户，我希望保存多组LLM配置并切换，以便不同场景使用不同模型 |
| P2 | 作为用户，我希望导出全部数据（碎片+故事）为JSON/Markdown，以便备份和迁移 |

### 2.5 搜索功能

> **设计动机**：碎片数量超过50条后，仅靠星系拖拽无法快速定位目标内容。搜索是碎片管理的必备基础设施。

| 优先级 | 用户故事 |
|--------|---------|
| P0 | 作为碎片较多的用户，我希望能通过关键词搜索碎片内容，以便快速找到某条特定灵感 |
| P0 | 作为用户，我希望搜索结果中高亮匹配的关键词，以便一眼看到匹配位置 |
| P1 | 作为用户，我希望按标签筛选搜索结果，以便在特定主题下缩小范围 |
| P1 | 作为组织型用户，我希望同时搜索碎片标题和故事标题，以便一站式查找 |
| P2 | 作为用户，我希望搜索历史自动保存，以便快速重复之前的搜索 |

**搜索入口**：星系Tab顶部固定搜索栏（默认收起，下滑展开）。

### 2.6 标签体系设计

> **设计动机**：纯自由标签会导致标签爆炸（50+个仅用一次的标签），需要结构化体系。

**标签分为两级**：

| 层级 | 类型 | 说明 | 示例 |
|------|------|------|------|
| **系统预设** | 灵感领域 | 碎片所属的大类，固定不可删除 | 科技、人文、艺术、生活、商业、自然 |
| **系统预设** | 内容形态 | 碎片的形式特征 | 观点、场景描写、人物设定、对话片段、问题 |
| **用户自定义** | 主题标签 | 用户自由创建，建议上限20个 | #赛博朋克 #AI伦理 #极简主义 |

**管理规则**：
- 用户自定义标签上限 **20个**（v1限制，防止标签爆炸）
- 支持标签**重命名**（全局生效，所有已标记碎片同步更新）
- 支持标签**合并**（将标签A合并到标签B，A自动删除）
- 碎片至少选择1个"灵感领域"标签，内容形态和自定义标签可选
- 标签在本地存储，不上传到服务器

| 优先级 | 用户故事 |
|--------|---------|
| P1 | 作为用户，我希望有预设的灵感领域标签可选，以便不用从零开始创建标签体系 |
| P1 | 作为用户，我希望重命名一个标签后所有关联碎片自动更新，以便修正命名不够准确的标签 |
| P1 | 作为用户，我希望能合并两个含义相近的标签，以便清理冗余标签 |
| P2 | 作为用户，我希望看到每个标签下有多少条碎片，以便了解自己的灵感分布 |

### 2.7 批量操作

| 优先级 | 用户故事 |
|--------|---------|
| P1 | 作为碎片较多的用户，我希望在星系中长按进入多选模式后批量删除碎片，以便清理不再需要的灵感 |
| P1 | 作为用户，我希望批量为选中碎片添加/修改标签，以便统一分类 |
| P2 | 作为用户，我希望批量归档碎片（隐藏但保留数据），以便清理星系视图但不丢失内容 |
| P2 | 作为用户，我希望按时间范围批量选择碎片（如"选择本周所有碎片"），以便快速批量操作 |

**交互设计**：长按任意星点 → 进入多选模式（节点出现选中框）→ 点击更多节点追加选择 → 底部弹出批量操作栏（删除 / 改标签 / 归档）。

## 四、UI/UX设计思路

### 4.1 视觉设计语言

**设计基调**：深邃 · 静谧 · 神秘 —— 模拟星空观测的沉浸体验

| 要素 | 设计决策 |
|------|---------|
| 主色调 | 深空黑背景 (#0A0A0F)，点缀星光白/紫/蓝渐变 |
| 辅助色 | 碎片星点：蓝白色系；故事星云：暖金色/紫色系；关联连线：半透明淡紫色 |
| 字体 | 中文：思源黑体；英文：Inter；标题适当使用有"宇宙感"的衬线字体点缀 |
| 图标 | 线性图标为主，星空/星座为主题的定制图标体系 |
| 动效 | 星点入场：淡入+缩放弹性动画；星系拖拽：惯性滑动；故事生成：打字机逐字输出 |

### 4.2 星系可视化设计（核心交互）

这是整个App最重要的设计挑战。核心设计思路：

**布局算法**：
- 采用**力导向图（Force-Directed Graph）**布局
- 碎片节点：根据标签相似度产生引力/斥力，同类标签的碎片自然聚拢形成"星座"
- 故事节点：位于关联碎片节点的几何中心偏上方，视觉上更大更亮（星云效果）
- 关联连线：碎片-故事之间半透明贝塞尔曲线连接
- 时间维度：Z轴映射到节点大小和透明度——越新的碎片越大越亮

**交互手势**：
| 手势 | 行为 |
|------|------|
| 单指拖动 | 平移星系画布 |
| 双指捏合 | 缩放星系（放大看局部星座，缩小看全局） |
| 单击星点 | 弹出碎片/故事详情卡片 |
| 长按星点 | 进入多选模式（为故事生成选素材） |
| 双击空白 | 重置到默认视角 |

**视觉层次**：
```
层4: 碎片详情/故事详情卡片 (浮层)
层3: 故事节点 (大、亮、暖色星云)
层2: 碎片节点 (小、蓝白星点)
层1: 关联连线 (半透明贝塞尔曲线)
层0: 深空背景 + 微尘粒子效果
```

### 4.8 错误状态与异常处理

| 错误场景 | 用户感知 | 错误信息 | 用户可执行操作 |
|---------|---------|---------|--------------|
| **LLM连接失败**（配置测试时） | Toast + 输入框标红 | "无法连接到 API 地址，请检查 URL 和网络连接" | 重新输入 / 查看帮助 |
| **LLM API Key无效**（配置测试时） | Toast + 输入框标红 | "API Key 验证失败（401），请检查 Key 是否正确" | 重新输入 |
| **故事生成中途失败** | 生成界面保留已生成内容 + 底部错误卡片 | "生成中断：请求超时。已生成的内容已保留。" | [重试] / [修改配置] / [保存当前内容] |
| **故事生成配额耗尽** | 错误卡片 | "API 返回：速率限制，请稍后再试" | [等待并重试] / [切换模型] |
| **故事上传失败** | Toast + 故事保持"未上传"状态 | "上传失败：网络不可用。故事已保存在本地，可稍后重试" | [重试上传] |
| **数据导出失败** | Toast | "导出失败：存储空间不足" | [清理空间] / [重试] |
| **首次登录无网络** | 全屏提示 | "首次使用需要网络连接以创建账户" | [检查网络设置] |

**错误处理原则**：
- 绝不丢失用户数据：生成中断时已产生的内容自动保存
- 区分可重试与不可重试错误，明确告知用户下一步
- 离线功能永远不受错误影响（碎片记录、星系浏览在任何网络状态下都可用）

### 5.5 星系布局算法核心思路

```
1. 初始化位置：按创建时间排列成螺旋状（银河系形态）
2. 力导向迭代：
   - 标签相似度 → 引力（标签相同的碎片相互吸引）
   - 碎片-故事关联 → 强引力（故事靠近其素材碎片）
   - 所有节点 → 中心引力（防止飘散过远）
   - 节点之间 → 斥力（防止重叠）
3. 每次新碎片加入：运行50轮迭代，动画过渡到新位置
4. 持久化：位置坐标存入Room，下次打开直接加载
```

### 5.8 数据迁移策略（Room Migration）

数据库schema一定会随版本迭代变化。必须在 Phase 0 就建立迁移框架：

```kotlin
// 迁移管理
@Database(
    entities = [FragmentEntity::class, StoryEntity::class, 
               LLMConfigEntity::class, UserEntity::class],
    version = 1,
    exportSchema = true  // 导出schema到文件，便于diff
)
abstract class AppDatabase : RoomDatabase() { ... }

// 示例：v1 → v2 迁移（新增fragment的isArchived字段）
val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(db: SupportSQLiteDatabase) {
        db.execSQL("ALTER TABLE fragments ADD COLUMN isArchived INTEGER NOT NULL DEFAULT 0")
    }
}

// 数据库构建时注册所有迁移
Room.databaseBuilder(context, AppDatabase::class.java, "astralog.db")
    .addMigrations(MIGRATION_1_2, MIGRATION_2_3, ...)
    .build()
```

**迁移原则**：
- 禁止使用 `fallbackToDestructiveMigration()`（生产环境数据丢失不可接受）
- 每次schema变更必须在CI中通过迁移测试（见5.9）
- 导出schema JSON文件纳入版本控制，每次PR需要review schema diff

### 5.11 大节点量星系聚合视图设计

> **触发条件**：当星系数总节点（碎片 + 故事）> 300 时，默认切换到聚合视图。用户可手动切换回详细视图（提示性能警告）。

**聚合算法**：
1. 按标签相似度将碎片聚类为"星座"（Constellation）
2. 每个星座显示为一个**发光的轮廓区域**（而非单个星点），内部星点数量越多，轮廓越大越亮
3. 星座名称 = 该聚类中出现最多的标签名
4. 故事节点保持独立显示（数量远小于碎片），不参与聚合

**视觉设计**：
```
详细视图 (≤300节点)              聚合视图 (>300节点)
┌──────────────────┐           ┌──────────────────┐
│   ·  ·    ✦      │           │                  │
│     ·  ·    ·    │           │   ┌─────────┐   │
│  ·   ✦    · ·   │    →     │   │ 科幻·12  │   │
│    ·   ·  ·      │           │   └─────────┘   │
│  ·     ·    ✦    │           │        ✦        │
│   · ·      ·     │           │   ┌──────┐      │
└──────────────────┘           │   │人物·8│      │
                               │   └──────┘      │
                               └──────────────────┘
```

**交互**：
- 点击星座轮廓 → 放大进入该星座的详细视图
- 双指缩放：聚合视图始终可见，放大到一定程度自动展开为详细视图
- 筛选（时间/标签）与聚合视图兼容

**周活跃创作者数**（每周至少生成1个故事的用户数）

这一定义同时衡量了"碎片记录"和"故事转化"两个核心价值行为，是产品健康度的综合体现。

### 7.3 L2驱动指标

| 指标 | 用途 |
|------|------|
| 日均碎片录入数/用户 | 衡量录入频次与产品融入日常的程度 |

### 7.4 关键漏斗


1. **配置LLM**这一步流失可能最大（技术门槛），需要在首次引导时提供"一键配置模板"降低门槛

## 九、推送通知与用户召回策略

> **已确认决策（Q8）：v1 纯本地定时提醒**，通过 WorkManager + NotificationManager 实现，无需 FCM 后端。用户卸载后推送即失效，v2 可升级为 FCM 服务端推送。

> **设计动机**：灵感工具天然适合温和提醒，但过度推送会适得其反。需要在"提醒价值"和"尊重用户"间找平衡。

### 9.1 通知类型

| 类型 | 触发条件 | 文案示例 | 优先级 |
|------|---------|---------|--------|
| **灵感提醒** | 连续3天未记录碎片 + 用户活跃时段（傍晚19-21点） | "今天的星空还空着——有什么想法值得记下来？" | P1 |
| **故事推荐** | 碎片>10条且7天未生成故事 | "你有15颗星在等待变成故事，要不要试试？" | P1 |
| **广场更新** | ~~故事广场有新故事~~（v1纯本地无法感知服务端变化，**v1不做此类推送**） | - | ~~P2~~ |
| **里程碑庆祝** | 第10/50/100条碎片，第1/10个故事 | "🎉 你已经记录了100条灵感！你的星系越来越壮阔了" | P2 |
| **数据备份提醒** | 每30天未导出数据 | "你已经一个月没备份了。建议导出数据以防丢失" | P2 |

### 9.2 推送策略规则

- **频率上限**：每周总计不超过3条推送
- **免打扰时段**：22:00 - 09:00 不发送任何推送
- **静默频道**：灵感提醒、故事推荐为"静默通知"（不出声音，只出现在通知栏）
- **首次请求权限**：在用户完成第5条碎片后弹窗请求（而非App首次打开），附带"我们承诺每周最多3条温和提醒"
- **可关闭**：每种通知类型可在设置中独立开关

### 9.3 深度链接

所有推送通知支持点击跳转：
- 灵感提醒 → 记录Tab
- 故事推荐 → 故事Tab（自动选中最近碎片）
- 里程碑 → 星系Tab（展示动画效果）

## 十一、冷启动内容策略

> **已确认决策（Q9）：不需要种子故事/KOL**，广场由首批用户自然填充。

### 11.1 问题诊断
故事广场上线时空无一物——用户打开后立刻离开，且不会回来。

### 11.2 v1 应对策略（无种子内容）

**策略一：降低广场的心理预期**
- 广场空状态设计强调"成为第一个"的荣誉感（见 4.7 空状态设计）
- 文案方向从"这里什么都没有"变为"你可能是第一个点亮这里的人"

**策略二：引导首批用户上传**
- 首次生成故事后，在保存环节增加轻引导："这是你的第1个故事——要不要分享到广场，让更多人读到？"
- 前10个上传到广场的故事，在"我的"页面获得"🪐 广场拓荒者"标记
- 广场显示实时用户故事总数（即使很少，也给社区感）

**策略三：广场可见性控制**
- v1 广场入口默认可见，但优先级低于核心创作流程
- 如果上线1个月后广场故事 < 5篇，考虑在 Phase 4.5 做一个小迭代——邀请3-5位内测用户集中创作几篇优质故事作为种子

**策略四：无需登录即可浏览**
- 未登录用户可预览广场（引流注册）
- 登录后才能上传和查看完整故事内容

## 十二、内容质量保障机制

### 12.1 碎片质量引导

| 机制 | 触发条件 | 行为 |
|------|---------|------|
| **碎片字数提示** | 碎片内容 < 15字 | 在记录页底部轻提示："细节越多，AI生成的故事越精彩"（非阻塞） |
| **素材不足警告** | 进入故事Tab时，选中的碎片总字数 < 50字 | 确认弹窗："选中的碎片内容较少，生成的故事可能不够丰富。是否继续？" [继续] [再选几条] |
| **碎片多样性建议** | 选中碎片来自同一标签 | 提示："都是'科幻'标签的碎片。试试加入不同角度的碎片，故事会更立体" |

### 12.2 生成质量兜底

- **Prompt Engineering**：System Prompt 中包含"如果素材不足以构成完整叙事，请生成散文/随笔风格而非强行编造剧情"
- **最低生成质量**：如果返回内容 < 200字（短篇模式），自动触发润色建议："故事有点短，要不要换个角度重新生成？"
- **生成失败保留**：即使生成中断，已输出的token自动保存为"草稿故事"，用户可手动续写

### 12.3 广场内容审核（v1轻量版）

| 级别 | 措施 |
|------|------|
| **自动过滤** | 后端关键词黑名单（违规词汇检测），命中则拒绝上传并提示 |
| **用户举报** | 故事详情页底部"🚩 举报"按钮 → 提交原因 → 累计3次举报自动隐藏 → 人工复核 |
| **上传频率限制** | 同一用户每小时最多上传2篇（防止刷屏/spam） |
