## Phase 2: 星系可视化（7-10 天）

> 目标：用户能在星图中看到自己的碎片，拖拽探索，这是产品的核心差异化体验。

### 2.1 力导向引擎（3 天）完成

| # | 任务 | 复杂度 | 验证标准 |
|---|------|--------|---------|
| 2.1.1 | `ForceDirectedEngine` — 核心算法实现（斥力 + 引力 + 中心力 + 阻尼） | 🔴 | 纯 Kotlin，无 Android 依赖 |
| 2.1.2 | 相似度计算：同 domainTag → 额外引力；同 formTag → 弱引力 | 🟡 | 同标签碎片聚合在一起 |
| 2.1.3 | 故事节点：固定引力源，关联碎片围绕在其周围 | 🟡 | 故事节点不动，碎片环绕 |
| 2.1.4 | 50 轮迭代 + 阻尼收敛 → 动画过渡到最终位置 | 🟡 | 碎片插入 3 秒内布局完成 |
| 2.1.5 | 算法单元测试：验证 10 节点布局收敛在 ±500 范围内 | 🟡 | `assertEquals` 坐标范围 |

### 2.2 Canvas 渲染（3-4 天）完成

| # | 任务 | 复杂度 | 验证标准 |

## 风险清单

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| Canvas 力导向性能不达标 | 中 | 高 | Phase 2 首日先做 500 节点 benchmark，不达标则引入 WebAssembly 渲染备选方案 |
| LLM API 供应商差异大 | 中 | 中 | Phase 3 先支持 OpenAI + DeepSeek 两个，再加通义千问 |
| 后端开发并行导致 Android 端阻塞 | 低 | 低 | Android 核心功能 100% 离线，不依赖后端 |
| 单人开发导致某 Phase 延期 | 高 | 中 | 每个 Phase 有明确的「可演示 MVP」，即使延期也能展示进展 |
| 国内网络环境导致 Firebase 不稳定 | 中 | 低 | 所有 Firebase 功能 fallback 为 no-op，不阻塞 App 运行 |
